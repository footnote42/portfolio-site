Squad Rotation Tool v2.0

Fair game‑time allocation for age‑grade rugby teams

I built the first version because I kept losing track of who’d played where. It worked—until it didn’t. Coaches forgot last week’s rotations, players moved positions, and I realised the tool needed a memory, not just a calculator.

Version 2.0 fixes that. It remembers what happened last time so I don’t have to.

What problem was I actually solving?

Rugby coaching at youth level is a balancing act:

  • Everyone should get fair minutes
  • Everyone wants to play their favourite position
  • Reality rarely cooperates

The old spreadsheet handled the maths but not the context. If a player missed a week, the system didn’t care. If someone played out of position, it forgot.

I needed something that behaved more like a coach’s notebook than a scheduling tool.

“Sometimes the best feature is just remembering what happened last time.”

How it works now

Stateful rotation engine

The tool stores last week’s lineup and uses it as a baseline. If a player didn’t attend, the system adjusts without punishing them the following week.

Position memory

Players who filled in at unfamiliar positions get rotated back to their preferred roles automatically. No more “why am I at prop again?” conversations.

Coach notes

A small notes panel lets me jot down context:

  • Injuries
  • Player swaps
  • Weather‑related chaos
  • The usual Saturday morning surprises

These notes feed into the next rotation cycle.

What I’d change next

I want to add a “rotation fairness meter” that shows who’s trending high or low on minutes. Not to optimise—just to keep things honest.

I’m also experimenting with a lightweight mobile view. Coaches don’t carry laptops onto muddy pitches.

Tech stack

Next.js — predictable routing, good DX
Supabase — simple persistence layer
TypeScript — probably too much, but it keeps me honest
Tailwind — fast iteration, easy theming with custom properties

The stack is intentionally boring. The interesting part is what the tool remembers, not how clever the architecture looks on a diagram.

Lessons learned

  • Tools get better when they reflect real behaviour, not ideal behaviour
  • Memory matters more than clever algorithms
  • “Good enough” becomes “actually useful” surprisingly quickly
  • If I build for my own Saturday chaos, others might find it helpful too

Try the tool

This isn’t a polished SaaS product. It’s a coach’s tool that happens to run in a browser. If you’re juggling youth squads and want something that remembers last week for you, this might help.
